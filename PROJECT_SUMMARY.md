# CareXR-2026 Project Summary

## Overview

CareXR-2026 is a **Snap Spectacles AR application** built in **Lens Studio** designed as a **gentle conversation coach for people with autism**. The system provides two real-time overlays on the wearer's AR display:

1. **Emotion detection** -- identifies the facial emotion of the person in front of the wearer (via DeepFace on a Python backend)
2. **Follow-up hints** -- suggests short, simple conversational nudges generated by GPT-4o-mini, based on what the other person just said and their detected emotion

Speech from "Person 2" (the non-wearer) is captured via a **lapel microphone** connected to the demo computer, transcribed by **Whisper**, and fed to the LLM alongside the current detected emotion. The hint is then displayed on the Spectacles as 3D text hovering above Person 2's head.

A reference project (**CueTips**) in the `Other Projects/` directory served as architectural inspiration.

### Data Flow

1. **Emotion path**: Spectacles camera frame → Base64 JPEG → ngrok → `/analyze` → DeepFace → emotion label → Spectacles display (left side)
2. **Hint path**: Lapel mic audio → adaptive silence detection → Whisper transcription → GPT-4o-mini (with current emotion) → hint stored → piggybacked on next `/analyze` response → Spectacles display (right side)

## Emotion Detection Pipeline

### On Spectacles (BackendEmotionDetector.js)

1. **Camera Init**: `CameraModule.createCameraRequest()` to access Spectacles camera
2. **Face Tracking**: `FaceFoundEvent` / `FaceLostEvent` (faceIndex 0)
3. **Frame Capture**: On each new frame (throttled to 0.1s), encodes as Base64 JPEG with `CompressionQuality.LowQuality`
4. **Backend Request**: POST to ngrok URL with `{ image: <base64> }` and `ngrok-skip-browser-warning` header
5. **Display Emotion**: Crossfades to the matching emotion bubble (Joy=pastel green, Sadness=pastel blue, Neutral=pastel beige) on the left side. All three bubbles overlap at the same position; only the active one is fully visible.
6. **Display Hint**: If response contains `follow_up_hint`, fades out the current hint bubble and fades in the new one (soft grey iMessage-style bubble, white text) on the right side.
7. **Face Lost**: All emotion bubbles and hint bubble fade out

### Backend (emotion_server.py)

#### Emotion Analysis (`/analyze` endpoint)
1. **Decode**: Base64 decode + downscale to max 480px
2. **Analyze**: `DeepFace.analyze()` with MTCNN detector backend
3. **Map**: Raw emotions mapped to categories:
   - `happy` → Joy, `sad` → Sadness
   - `angry`, `neutral`, `disgust`, `fear`, `surprise` → Neutral
4. **Store**: Current emotion saved to shared state (used by GPT for context)
5. **Response**: `{ emotion, confidence, label, latency_ms, follow_up_hint? }`

#### Speech-to-Hint Pipeline (background mic thread)
1. **Calibration**: On startup, records 4 seconds of ambient noise to compute baseline RMS
2. **Adaptive Threshold**: Speech threshold set to `ambient_rms * 2.5` (minimum floor of 300)
3. **Capture**: `sounddevice.InputStream` reads audio in 0.5s chunks from lapel mic
4. **Silence Detection**: When RMS drops below threshold for 1.5s, the buffered audio is a complete sentence
5. **Safety Valve**: If someone talks for 15s straight without pausing, sends anyway
6. **Transcribe**: Whisper (base model) transcribes the PCM16 audio
7. **Generate Hint**: GPT-4o-mini receives Person 2's transcript + current emotion, returns `{"follow_up_hint": "..."}`
8. **Deliver**: Hint is stored in shared state and piggybacked on the next `/analyze` response

#### GPT System Prompt
The LLM is prompt-engineered as a "gentle conversation coach" that:
- Provides one short sentence (under 15 words)
- Uses warm, encouraging, simple language
- Frames suggestions gently ("Maybe ask about...", "You could say...")
- Matches the emotional tone of Person 2
- Returns structured JSON: `{"follow_up_hint": "..."}`

## How to Run

### 1. Set Up Environment

```powershell
cd Backend
pip install -r requirements.txt
```

Edit `Backend/.env` with your OpenAI API key:
```
OPENAI_API_KEY=sk-your-key-here
```

### 2. Start the Backend

```powershell
cd Backend
python emotion_server.py
```

Expected startup output:
```
============================================================
CareXR Backend Server
============================================================

[1/3] Loading Whisper model (base)...
  Whisper ready.

[2/3] Warming up DeepFace (mtcnn)...
  DeepFace ready.

[3/3] Starting lapel mic listener...
  Available audio devices:
  [... device list ...]
  Using default input device (set MIC_DEVICE_INDEX in .env to override)
[Mic] Calibrating ambient noise (4s — don't talk into the mic)...
[Mic] Ambient RMS: 150 -> speech threshold: 375
[Mic] Listening for Person 2 (threshold=375)...

Server ready on 0.0.0.0:5001
```

**Important**: During the 4-second calibration, don't talk into the lapel mic. Let it capture ambient room noise so it can set the right threshold.

### 3. Start ngrok Tunnel

```powershell
ngrok http 5001
```

Note the `Forwarding` URL (e.g., `https://xxx.ngrok-free.dev`).

### 4. Configure Lens Studio

In Lens Studio, select `BackendEmotionController` and set `backendUrl` to:
```
https://YOUR-NGROK-URL.ngrok-free.dev/analyze
```

### 5. Select Lapel Mic (if not default)

Run the server once to see the device list. Find your lapel mic's index number, then set in `.env`:
```
MIC_DEVICE_INDEX=2
```

### 6. Push to Spectacles

Use Lens Studio's "Push to Device" to deploy the lens.

### 7. Kill/Restart the Backend

```powershell
Get-Process python -ErrorAction SilentlyContinue | Stop-Process -Force
python emotion_server.py
```

## Troubleshooting

| Symptom | Cause | Fix |
|---------|-------|-----|
| `Backend error (0):` on Spectacles | Backend not running or ngrok disconnected | Restart `emotion_server.py` and verify ngrok is forwarding to port 5001 |
| `Backend error (502):` on Spectacles | ngrok running but backend is not | Start `python emotion_server.py` and wait for "Server ready" |
| `Backend error (500): float32 not JSON serializable` | Numpy types in DeepFace output | Fixed with `NumpySafeEncoder` class |
| `Exception: fetch method moved to InternetModule` | Wrong Lens Studio module | Use `require('LensStudio:InternetModule')` |
| Emotions always "Neutral" on Spectacles | Blend shapes return near-zero on device | Use BackendEmotionDetector, not EmotionDetector |
| Flask server becomes unresponsive | Flask dev server deadlocks | Switched to `waitress` production server |
| `Camera init failed` in preview | CameraModule is Spectacles-only | Expected in preview mode |
| `Base64.encodeTextureAsync` fails silently | MediumQuality compression | Use `CompressionQuality.LowQuality` |
| Mic picks up crowd noise | Ambient noise floor too close to speech | Adaptive calibration handles this; don't talk during the 4s calibration period |
| No hints appearing on glasses | Backend mic not capturing or OPENAI_API_KEY not set | Check `/mic/status` endpoint and verify `.env` has a valid API key |
| Whisper transcribes noise as speech | Silence threshold too low | Increase `NOISE_MULTIPLIER` in `emotion_server.py` (default 2.5) |

## Design Decisions & Lessons Learned

1. **Blend shapes don't work on Spectacles**: Face mesh blend shape weights worked in preview but returned near-zero on hardware. Forced pivot to backend DeepFace approach.

2. **DeepFace over LLM for emotions**: Uses DeepFace (local ML) instead of an LLM for direct emotion classification -- avoids API costs and reduces latency.

3. **waitress over Flask dev server**: Flask development server deadlocked under sustained Spectacles load. waitress resolved this.

4. **MTCNN detector backend**: Better accuracy than OpenCV's Haar cascades, with acceptable speed (~300ms per analysis).

5. **Lapel mic over Spectacles mic**: Spectacles microphones are wearer-focused and don't reliably capture bystander speech. A separate lapel mic on Person 2 gives clean audio even in loud rooms.

6. **Adaptive noise calibration**: Instead of a fixed volume threshold, the system measures ambient noise for 4 seconds at startup and sets the speech threshold relative to the room's noise floor. This makes it work in both quiet and loud (200+ person) environments.

7. **Piggybacking hints on /analyze**: Rather than adding a separate polling endpoint for hints, the follow-up hint is attached to the next `/analyze` response. This avoids extra network calls from the Spectacles and simplifies the client-side code.

8. **One-shot hint delivery**: Hints are consumed (cleared) when read, so the same hint doesn't repeat on every frame. New hints from GPT trigger a fade-out/fade-in transition.

9. **Processing lock for Whisper**: A `threading.Lock` prevents multiple concurrent Whisper transcriptions, which would thrash the CPU and produce garbled results.

10. **Safety valve buffer (15s)**: If Person 2 talks continuously without pausing, the system sends the audio after 15 seconds anyway, preventing unbounded memory growth.

11. **ngrok-skip-browser-warning header**: Required on all requests to free ngrok URLs to bypass the interstitial page.

12. **LowQuality compression**: `Base64.encodeTextureAsync` with `MediumQuality` silently fails on Spectacles. `LowQuality` works reliably (47-84 KB payloads).

13. **Color-coded emotion bubbles with crossfade**: Three overlapping `Text` bubbles (Joy=pastel green, Sadness=pastel blue, Neutral=pastel beige) at the same position. The script animates alpha to crossfade between emotions — the active one fades in while others fade out. Recycled from CueTips' text bubble UI pattern. Each has a world-space `Canvas` for rendering under Head Binding.

14. **iMessage-style hint bubble**: The LLM hint uses a soft grey background with white text (similar to Apple iMessage received messages). When a new hint arrives, the old bubble fades out, text swaps, and the new bubble fades in.

## Reference: CueTips Project

The `Other Projects/CueTips/` directory contains a separate Snap Spectacles project that served as reference for:

- **ngrok tunneling pattern**: Exposing a local backend to Spectacles
- **VoiceML transcription**: On-device ASR (not used in CareXR due to bystander speech limitations)
- **Backend architecture**: Flask server with image/audio processing endpoints
- **Remote fetch pattern**: Using `InternetModule` for HTTP from Lens Studio

Key difference: CueTips uses Google Gemini (LLM) for emotion analysis, while CareXR uses DeepFace (local ML) for emotions and GPT-4o-mini only for conversation hints.

Additionally, CareXR recycled CueTips' **text bubble UI pattern** (Text component with Background enabled, world-space Canvas) for displaying the emotion label and LLM hint as rounded bubbles near the tracked head.
